My Experience With Lisp

03.04.2022 04:32

When I worked on web applications in a previous job, I had to maintain a lot of unfamiliar systems using old libraries. Often times I would need to experiment with them to understand how they worked. The web applications were large and took a long time to recompile, so tweaking the existing setup in a development environment wasn’t a good option. I would usually end up building an individual sandbox application for each library to test it in isolation. It worked, but felt cluttery and unpleasant.

Many of our webapps had a substantial amount generic logic that was wired up with a large number of individual settings in massive configuration files. This let developers change our libraries’ behavior at runtime without recompiling the application, but it made the libraries very hard to understand and made applications more brittle.

These were just things that I came to accept as “how enterprise development is done”, and I didn’t really enjoy it, although I appreciated that it worked decently enough.

A few years ago, I downloaded Emacs on a whim. I was bored and using a Windows desktop hooked up to my big screen tv. I opened up the application and saw a manual on the splash screen, so I figured why not actually read the manual for once in my life?

The manual had a bunch of sections, including an “introduction to Emacs Lisp”. I had never heard of Lisp before that point. Following along with the manual felt like playing a weird Zachtronics game. Luckily I enjoy those games, even if I suck at them, and I was interested enough to keep at it. I ended up spending a decent amount of time learning the editor and Emacs Lisp, then learning Common Lisp, diving into Clojure, trying out Racket, checking out Gerbil, and hacking around in Guix using Guile.

To me, the power of Lisp is that it makes it very easy to build flexible, robust software without excessive boilerplate, config files, and the need for lengthy recompilation. S-expressions and macros are a very intuitive way to build abstractions and build code as a set of pure functions that you compose and thread state through at the highest level possible. While this is doable in any language, I think it’s harder to focus on while juggling with noisy syntax and verbose language idioms. Just try comparing an implementation of a scheme written in scheme to the implementation of say, Clojure, written in Java, and you’ll see what I mean.

Most importantly, it’s shown me that I actually enjoy software development, which isn’t something I would have confidently said before. I’ve largely moved on from Lisps since, but I think the time I spent learning them was well spent.
