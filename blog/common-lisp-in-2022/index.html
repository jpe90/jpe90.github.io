Overview of Common Lisp

04.04.2022 14:15

In this post I’ll give my thoughts on the strengths and weaknesses of programming in Common Lisp.

CL is particularly well suited for building applications which can be inspected and modified as they run. Applications can be running locally, on a server, or even on embedded devices. CL’s macros empower you to cut down substantially on boilerplate and create useful DSLs, which can sound very abstract, but in practise it just means that you get the ability to shape the language to the problem you’re working on far more than you can with most other languages. What makes Common Lisp unique even among other Lisps is how well it does both of these.

If you’ve been considering investing some time in learning CL, you might want to know it’s weaknesses. An interesting thing about CL is that it’s biggest weaknesses are social, not necessarily technical. The power to shape the language is often used by library authors in ways you wouldn’t expect, which can lead to surprising situations where you can’t really predict how the code you’re interfacing with works. (As a side note, this drawback was very influential in the design of Clojure in its efforts to encourage idiomatic program design and limit the use of reader macros to radically change the langage).

As an example: it’s not uncommon exploit mutability under the hood for performance reasons. I’ve had a few insidious bugs creep into programs where my program state would change in places where I wasn’t expecting it to because I call into a function which mutates the library’s internal state without any indication it was doing so.

Some people are turned off by how many libraries don’t see tons of updates. In some cases this is due to the language’s stability. CL code is not subject to the same level of bitrot as other ecosystems, and since the language standard hasn’t changed in an eternity, you don’t get the same situation where the language evolves and breaks existing code.

Of course, this doesn’t apply for code that makes heavy use of FFI (for GUI development, for example). If CL code is leaning on bindings to another language, and that language’s code changes, the CL bindings need to be updated in lockstep or they will break. This is where you really feel the language’s relative unpopularity, especially if you’re trying to build software on uncommon arcitechtures that aren’t as well supported- on my ARM mac, QT and GTK bindings fail to build.

Before I end this post I should note that Common Lisp in Emacs is the most fun I’ve ever had programming. At the same time, for a real-world project, I would almost never use it these days. The more experienced I’ve gotten with programming, I’ve increasingly come to value languages with strong network effects, that can produce fast code and small binaries without insane arm twisting. Common Lisp can be quite fast, but it falters on the other two. Still, I would definitely recommend dipping your toes in the water if you’re on the fence about learning it.
