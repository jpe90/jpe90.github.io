Fine-Tuning NLP Transformers for Task Automation

This is a WIP early draft that I'm sharing to get feedback as I write it. If you have any, please shoot me an email.

Introduction

In a previous post, I explored using LLMs to automate tasks that involved determining how to categorize strings of text. In that post, I used a wrapper around llama.cpp, which ran inference on the 8B parameter, 4-bit quantized version of Llama 3. The number of parameters are a measure of a model's capability; for comparison, GPT-4o, powering ChatGPT has 1.7 trillion parameters. With Llama 3 I got ~92% accuracy for my task out of the box, which bumped up to ~95% with some post-processing.

I didn't have enough knowledge or intuition to know whether that performance was any good relative to the capabilities of an 8B model. My mental model of 8B models at the time was "ChatGPT with severe limitations", which primed me to be happy with whatever I could get. But thinking about it a little more deeply, I couldn't help but wonder how much performance budget I was ignoring with this setup.

One possible option would be to train a model completely from scratch using nothing but training data. I originally measured performance with 30 strings of text that were classified into one of three categories: general feedback, order inquiries, or refund requests. If I massively expanded the amount of training data, I wondered if I could reach an amount that would allow me to train a tokenized transformer that is solely designed to classify its input. In theory, this could give me an extremely efficient network that is capable of carrying out this task with the smallest possible size.

I ended up not pursuing this option because I worried that it would be prohibitively difficult for me to supply enough training data to be able to correctly classify whatever weird wording might be thrown at it. I suspect it takes quite a lot of text for the attention mechanism of a deep learning network to be able to handle nuances of the English language without getting confused. Training from scratch, while an interesting idea, seemed like too ambitious of a task for what I was looking to do. It makes sense to me that a huge part of the magic of neural networks is compiling a well-rounded and complete dataset, and as an amateur, the likelihood that I would do so did not seem particularly high.

I decided instead to explore fine-tuning an existing model. The Llama 3 model, trained on a representative sample of the entire internet and designed to answer any question, seemed like overkill as a starting point. I didn't need something that could talk for hours about nuanced details of the theory of relativity in order to classify my text strings. I sought out other smaller models that might be a more sensible starting point.

I found a model "Bidirectional Encoder Representations from Transformers", also known as BERT, introduced in October 2018 by researchers at Google. It has very similar underlying architecture to Llama 3 or GPT-o, but was trained on much less data. It has an even smaller cousin DistilBERT which uses 40% fewer parameters, runs 60% faster, and preserves 95% of the performance of BERT. DistilBERT has a total of 66 million parameters and was trained on the Toronto Book Corpus and English Wikipedia.

DistilBERT Architecture Overview

DistilBERT, like its larger counterpart BERT, is based on the Transformer architecture. At its core, DistilBERT consists of several key components that work together to process and understand text. Text input is first passed through embedding layer. This layer is responsible for converting input tokens (which are essentially words or parts of words) into dense vector representations. These vectors capture the semantic meaning of the words in a format that the model can work with.

Once the text is embedded, it passes through multiple Transformer blocks. These blocks contain layers of self-attention and feed-forward neural networks. The self-attention mechanism allows the model to weigh the importance of different words in the input. This means the model can understand context and relationships between words, much like how we understand language by considering words in relation to each other. After the self-attention layer, feed-forward neural networks further process the attention output, allowing the model to capture complex patterns in the data.

Fine-tuning Process

The fine-tuning process involves adjusting the pre-trained DistilBERT model to our specific classification task.

Data Preparation

We start by loading our labeled data:

def load_data(file_path):
    with open(file_path, 'r') as f:
        return [line.strip() for line in f]

feedback = load_data('data/feedback.txt')
inquiries = load_data('data/inquiries.txt')
refunds = load_data('data/refunds.txt')

all_texts = feedback + inquiries + refunds
all_labels = [0] * len(feedback) + [1] * len(inquiries) + [2] * len(refunds)

This code loads our three categories of text and assigns numerical labels to each. We're simply reading text files and creating lists of our input texts and their corresponding labels.

Dataset and DataLoader

We create a custom dataset class to handle tokenization:

class TextClassificationDataset(Dataset):
    def __init__(self, texts, labels, tokenizer, max_len):
        self.texts = texts
        self.labels = labels
        self.tokenizer = tokenizer
        self.max_len = max_len

    def __getitem__(self, item):
        text = str(self.texts[item])
        label = self.labels[item]

        encoding = self.tokenizer.encode_plus(
            text,
            add_special_tokens=True,
            max_length=self.max_len,
            return_token_type_ids=False,
            padding='max_length',
            truncation=True,
            return_attention_mask=True,
            return_tensors='pt',
        )

        return {
            'text': text,
            'input_ids': encoding['input_ids'].flatten(),
            'attention_mask': encoding['attention_mask'].flatten(),
            'labels': torch.tensor(label, dtype=torch.long)
        }

A tokenizer is a tool that breaks down text into smaller units called tokens. These could be words, parts of words, or even punctuation. The tokenizer also handles converting these tokens into numbers that the model can understand. We're using the DistilBertTokenizer because we want to use the same tokenizer that the model was trained with. Under the hood, it uses WordPiece subword segmentation. This is a little older than alternatives like SentencePiece of BPE.

We're using the encode_plus method of this tokenizer. This method does several things: It tokenizes our input text, adds special tokens that DistilBERT expects (like [CLS] at the start and [SEP] at the end), pads or truncates our input to a specified maximum length, and creates an "attention mask" which tells the model which tokens are actual input and which are padding.

The torch Dataset is a PyTorch class that represents a dataset. By inheriting from this class, we're creating a custom dataset that PyTorch can easily work with. The __getitem__ method is called when we want to access an item from our dataset.

We use PyTorch's DataLoader to efficiently batch and shuffle our data:

train_dataloader = DataLoader(train_dataset, batch_size=16, shuffle=True)
test_dataloader = DataLoader(test_dataset, batch_size=16)

The DataLoader is a PyTorch utility that helps manage batching of data and provides an iterable over the Dataset. For our training data, we set shuffle=True, which means it will randomly shuffle the data at the start of each epoch. This shuffling helps prevent the model from learning any unintended patterns based on the order of the training data.

We use a batch size of 16, meaning the DataLoader will yield batches of 16 examples at a time. This batch size is a balance between memory usage and training speed. For the test data, we don't need to shuffle, so we omit the shuffle parameter.

During training and evaluation, we'll iterate over these DataLoaders, which will give us batches of data in the format our model expects. This abstraction simplifies our training loop and makes it easier to work with large datasets that might not fit into memory all at once.

Train-Test Split

Before we start training, we split our data into training and testing sets:

def train_test_split(data, test_size=0.1):
    split_index = int(len(data) * (1 - test_size))
    return data[:split_index], data[split_index:]

train_data, test_data = train_test_split(list(zip(all_texts, all_labels)))
train_texts, train_labels = zip(*train_data)
test_texts, test_labels = zip(*test_data)

We're using a manual split where the first 90% of the data is used for training and the last 10% for testing. I opted to do it this way so that I could make sure all of the samples that I evaluated in the previous blog post were tested by this model without it having seen them during training. This way, I could get a better sense of their relative performance. More on that later.

Training Loop

The training process involves iterating over our data multiple times and updating the model's parameters to minimize the classification error:

def train_epoch(model, data_loader, optimizer, device):
    model.train()
    total_loss = 0
    total_correct = 0
    total_samples = 0

    for batch in data_loader:
        input_ids = batch['input_ids'].to(device)
        attention_mask = batch['attention_mask'].to(device)
        labels = batch['labels'].to(device)

        optimizer.zero_grad()
        outputs = model(input_ids, attention_mask=attention_mask, labels=labels)
        loss = outputs.loss
        logits = outputs.logits

        _, predicted = torch.max(logits, 1)
        total_correct += (predicted == labels).sum().item()
        total_samples += labels.size(0)

        loss.backward()
        optimizer.step()

        total_loss += loss.item()

    avg_loss = total_loss / len(data_loader)
    accuracy = total_correct / total_samples
    return avg_loss, accuracy

This function performs one training epoch, updating the model's parameters and calculating the loss and accuracy.

Model Configuration and Training

tokenizer = DistilBertTokenizer.from_pretrained('distilbert-base-uncased')
model = DistilBertForSequenceClassification.from_pretrained('distilbert-base-uncased', num_labels=3)

train_dataset = TextClassificationDataset(train_texts, train_labels, tokenizer, max_len=128)
test_dataset = TextClassificationDataset(test_texts, test_labels, tokenizer, max_len=128)

First, we initialize our tokenizer and model. We're using the pre-trained 'distilbert-base-uncased' model, which means it's been trained on lowercase English text. The num_labels=3 parameter tells the model we're dealing with a three-class classification problem.

We set max_len=128 when creating our datasets. This maximum length parameter determines the longest sequence of tokens our model will process. Shorter sequences will be padded, and longer ones will be truncated.

The batch size of 16 in our DataLoader means we'll process 16 examples at a time during training. This is another balance between memory usage and training speed. Larger batch sizes can lead to faster training but require more memory.

device = torch.device('mps' if torch.backends.mps.is_available() else 'cpu')
model.to(device)

Next, we move our model to the appropriate device. We're checking if Apple's Metal Performance Shaders (MPS) are available, which can accelerate training on compatible Mac hardware (I just did it this way because I'm running on my Macbook, you would want to change this if you wanted to run it optimally on different hardware).

optimizer = AdamW(model.parameters(), lr=2e-5)

Neural networks rely on nonlinear activation functions to model complex relationships in data. These functions, like tanh or ReLU, work best when their inputs are well-distributed across their operating range. For instance, a tanh function, which maps inputs to outputs between -1 and 1, is most effective when its inputs span this entire range rather than being concentrated at the extremes.

Ensuring this ideal distribution of inputs became a significant challenge as networks grew deeper. In 2015, researchers introduced Batch Normalization as a solution. This technique normalizes the inputs to each layer, improving the flow of data through the network. This worked, but it also introduced complexities.

The search for better optimization techniques led to the development of adaptive learning rate methods like Adam (Adaptive Moment Estimation) and later the variant AdamW. To be honest, I don't know much about these methods other than the fact that they solve this issue (among others) and that they see widespread use. Learning more about them is on my bucket list.

We're setting a learning rate of 2e-5 (0.00002). It's small enough to allow for fine adjustments to the pre-trained weights without causing drastic changes that could destroy the model's pre-trained knowledge. Too high, and your model might overshoot optimal solutions; too low, and your model might train too slowly or get stuck in suboptimal solutions.

num_epochs = 5
for epoch in range(num_epochs):
    print(f'Epoch {epoch + 1}/{num_epochs}')
    train_loss, train_acc = train_epoch(model, train_dataloader, optimizer, device)
    print(f'Train loss {train_loss:.4f} accuracy {train_acc:.4f}')

    val_loss, val_acc = evaluate(model, test_dataloader, device)
    print(f'Val loss {val_loss:.4f} accuracy {val_acc:.4f}')

torch.save(model.state_dict(), 'model/distilbert_classifier.pth')
print("Model saved successfully.")

The training loop runs for 5 epochs. An epoch is one complete pass through the entire training dataset. This was a YOLO pick, and judging from the results it might be one or two more than we needed.

In each epoch, we train the model on the training data and then evaluate it on the validation (test) data. We print out the loss and accuracy for both training and validation sets. The .4f formatting in the print statements is a Python string formatting specifier that displays floating-point numbers with 4 decimal places. This gives us a consistent, readable format for our metrics.

After training, we save our model using torch.save(). This function saves the model's state_dict. A state_dict in PyTorch is a Python dictionary that maps each layer to its parameter tensors. It contains all the learned weights and biases of our model. By saving the state_dict, we're essentially saving all the knowledge our model has gained during training.

We save our model with a .pth file extension. This is a common convention in PyTorch for saved model files, standing for "PyTorch". The .pth file is essentially a serialized version of our model's state_dict. This contains all the weights and biases for the network.

To use this saved model later, we would load it like this:

model = DistilBertForSequenceClassification.from_pretrained('distilbert-base-uncased', num_labels=3)
model.load_state_dict(torch.load('model/distilbert_classifier.pth'))
model.eval()

Results and Discussion

After fine-tuning, I got an accuracy of 100% on the test set. This needs to be taken with a few grains of salt:

  1. The task was extremely easy by design. It reflected a very doable classification task that reflects a number of real-world situations I've run into.
  2. The training and test data does not come from real-world data, and as such it does not reflect the weirdness and oddities that you would get from real people.

Because of these factors, achieving 100% accuracy doesn't necessarily mean that much. But there are a few things worth noting. For one, that 100% accuracy includes the entirety of the test set that Llama 3 8B only achieved 92% on, with additional test data added in. And just for kicks, I later tried coming up with weird and contrived strings of text to try to confuse the model. I could not get it to incorrectly classify anything.

For simplicity, I ran with the assumption that each string of text had one and only one classification. It would be much more useful if each string of text instead could have 0-3 classifications. This would also almost certainly bring down the accuracy of the model because determining whether a given string belongs to a single category at all seems like it would be a much more complex task than what it had to learn during training.
